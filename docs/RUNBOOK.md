# Operations runbook

This runbook covers routine operation, health checks, incident response, backup, and upgrade procedures for the GhostMCP alpha runtime.

## Service health

For a systemd deployment:

```bash
sudo systemctl status ghostmcp
sudo journalctl -u ghostmcp --since "15 minutes ago"
```

For a container:

```bash
docker ps --filter name=ghostmcp
docker logs --since 15m ghostmcp
```

At the MCP layer, use:

- `runtime_probe_tool` for readiness and liveness information
- `server_health_tool` for policy and toolchain summary
- `toolchain_status_tool` for installed, missing, enabled, and disabled integrations
- `metrics_tool` for call, failure, timeout, and denial counters

Do not expose detailed health output to untrusted clients. It contains useful operational and policy information.

## Startup verification

After every deployment or restart:

1. Confirm the process is running as the dedicated non-root account.
2. Confirm the expected transport and bind address.
3. Verify remote authentication rejects a request without credentials.
4. Verify the dashboard rejects unauthenticated access if enabled.
5. Confirm the SQLite database is writable.
6. Confirm the audit sink is writable and the hash chain continues.
7. Review the toolchain status for unexpected binaries or missing approved binaries.
8. Verify raw wrappers and plugins match their allowlists.
9. Execute one passive test inside a dedicated test engagement.
10. Review logs and the audit event generated by the test.

## Routine checks

Daily or before an engagement:

- Review disk usage for database, audit logs, scanner output, and uploaded files.
- Confirm the audit signing key is readable only by the service account or root group.
- Check for failed or stuck scans.
- Review scheduled scans due within the next day.
- Confirm engagement scope has not expired or changed.
- Check package and container advisories.
- Verify the system clock is synchronized because schedules use UTC.

Weekly:

- Back up the SQLite database.
- Copy audit logs to separate storage.
- Review raw-wrapper and plugin allowlists.
- Remove expired engagements and unused credentials according to retention policy.
- Rebuild the container or virtual environment from current locked dependencies.

## Logs and audit data

Application logs describe runtime behavior. The audit JSONL sink records guarded calls and chain metadata. They serve different purposes and should be retained separately.

Recommended paths:

```text
/var/log/ghostmcp/service.log
/var/log/ghostmcp/audit.jsonl
/etc/ghostmcp/audit-hmac.key
/var/lib/ghostmcp/ghostmcp.db
```

Protect the audit key separately from the audit log. A local attacker who can modify both can reconstruct a valid-looking chain.

When shipping logs to a SIEM:

- Preserve event order.
- Keep the original JSON event intact.
- Record host, service instance, and ingestion time outside the signed event.
- Alert on chain verification failures, authorization denials, repeated timeouts, and unexpected tool registration.

## Database backup

The safest basic procedure is to stop writers before copying the database.

```bash
sudo systemctl stop ghostmcp ghostmcp-dashboard 2>/dev/null || true
sudo cp --preserve=mode,ownership,timestamps \
  /var/lib/ghostmcp/ghostmcp.db \
  /var/backups/ghostmcp/ghostmcp-$(date -u +%Y%m%dT%H%M%SZ).db
sudo systemctl start ghostmcp
```

If WAL mode has active writers, copying only the main database file can produce an incomplete backup. Stop writers, use SQLite's backup facility, or copy the database together with its `-wal` and `-shm` files.

Test restores periodically on a separate host.

## Audit verification

Validate:

- The first event starts from the expected chain origin.
- Each `prev_hash` matches the preceding `event_hash`.
- Canonical event serialization reproduces the stored hash.
- HMAC signatures verify with the expected key.
- There are no unexplained gaps or duplicate sequence points.

A verification failure is an incident. Preserve the files, stop rotation or cleanup jobs, and investigate before restarting normal operation.

## Failed scans

For a failed scan:

1. Read the persisted error and service logs.
2. Confirm the target remains in engagement scope.
3. Confirm the requested tool level is allowed globally and by the engagement.
4. Verify required binaries are installed and allowlisted.
5. Confirm credential injection did not require a disabled or unavailable backend.
6. Check process timeout and output limits.
7. Verify filesystem paths are inside `GHOSTMCP_ALLOWED_FILE_ROOTS`.
8. Retry only after identifying whether the failure was environmental, policy-related, or tool-related.

Do not loosen global policy just to make one failed scan run. Create a narrowly scoped engagement or adjust the specific allowlist after review.

## Stuck or queued scans

The dashboard worker is in process. If the process exits, an item already queued in memory is not restored automatically.

After restart:

- Inspect the persisted scan status.
- Confirm no external scanner process remains running.
- Mark or document abandoned work according to the operational process.
- Resubmit only after checking for side effects from the previous attempt.

## Credential rotation

When rotating local encrypted credentials:

1. Stop GhostMCP.
2. Back up the encrypted store and salt file.
3. Install the new key file with mode `0600` or equivalent restricted access.
4. Re-encrypt or recreate stored credentials under the new key.
5. Start the service and test retrieval with a non-destructive operation.
6. Destroy old key material according to policy.

For Vault, AWS, or GCP, rotate the provider secret and service identity according to the provider's normal process. Avoid long-lived cloud access keys.

## Transport-token rotation

1. Generate a long random replacement token.
2. Update the server secret source.
3. Update authorized clients.
4. Restart or reload the server.
5. Confirm the old token fails.
6. Confirm the new token succeeds.
7. Review logs for unexpected use of the old token.

Use different tokens for the MCP gateway and dashboard.

## Incident response

If compromise or unauthorized execution is suspected:

1. Stop the MCP gateway and dashboard.
2. Preserve database, audit, application logs, service configuration, and process information.
3. Revoke transport tokens and external secret-manager credentials.
4. Disable raw wrappers and plugins.
5. Block network access at the firewall.
6. Verify the audit chain and identify the first suspicious event.
7. Review engagement scope, registered tools, child processes, and filesystem changes.
8. Rebuild from a trusted commit and locked dependencies rather than cleaning the existing environment in place.
9. Restore only verified database content and rotate all keys.
10. Document affected targets and notify engagement owners.

Do not delete evidence during containment.

## Upgrade procedure

1. Read the release notes and compare `.env.example` with the deployed environment.
2. Back up database and audit data.
3. Stop services.
4. Fetch the intended commit or tag.
5. Recreate the virtual environment or rebuild the container from locks.
6. Run Ruff, mypy, Bandit, dependency audit, tests, and package build.
7. Start one instance.
8. Verify database access, audit continuity, authentication, tool registration, and one passive workflow.
9. Start the dashboard only after the gateway is healthy.
10. Monitor logs and metrics closely after the upgrade.

The current release is alpha. Do not assume configuration or database behavior remains unchanged between alpha versions.

## Rollback

Rollback requires both code and compatible state:

1. Stop services.
2. Preserve the failed upgrade state for analysis.
3. Restore the previous code or image.
4. Restore the pre-upgrade database if the newer version changed state incompatibly.
5. Restore matching configuration and lock files.
6. Verify authentication and audit continuity before reopening access.

Never point an older version at a database that may contain unsupported schema changes without testing first.
